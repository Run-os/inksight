## 新项目核心功能规划（两大功能）
- 产出技术规划文档：`docs/core-features-plan.md`（v1，待评审）。
- 两大核心功能：① 原生待办列表（设备本地渲染）；② 网络图片展示（后端下发 1-bit BMP，本地缓存≤5张，按键切换）。

## 已与用户确认的三项决策
- D1 待办来源：Web/App 后台管理，设备只拉取展示（设备无键盘/触控）。
- D2 图片后端：独立新建 FastAPI 服务 `inksight-content`（与现有 LLM backend 解耦）。
- D3 按键模型：短按 KEY(GPIO18)=上一页/上一张；短按 BOOT(GPIO0)=下一页/下一张。
  - 规划建议（待确认）：长按 KEY(≥2s)=在待办/图片间切 App；长按 BOOT(≥2s)=进配网 Portal（原长按 KEY 行为迁移）。
  - BOOT(GPIO0) 仅在"按住+重新上电"进下载模式，运行时长按可安全复用为功能键。

## 关键事实缺口（重要）
- 现有固件 `display.cpp::drawText` 仅内置 5×7 ASCII 字体，**无 CJK 字库**，无法渲染中文待办正文。
- 结论："待办原生渲染"必须在固件新增 CJK 16×16 点阵字库（建议 3500 常用字 ~112KB，可放 Flash/PSRAM）+ `draw7Seg` 数码管时间 + `drawMixed` 中英文混排。
- 图片模式：后端把图预处理成内容区尺寸(400×214)的 1-bit BMP 下发，设备原生画 chrome 并 blit 图片缓冲，本地缓存 5 张用 PSRAM。

## 字库原料与精确点阵格式（已探明，2026-07-13）
- 仓库已含可直接用作字库原料的中文矢量/位图字体，无需外找：
  - TTF（首选）：`backend/fonts/truetype/NotoSerifSC-Regular.ttf`（思源宋体，完整中文，可精确栅格到 16×16）。
  - PCF 位图（备选）：`backend/fonts/bitmap/wqy-zenhei-13.pcf`、`NotoSerifSC-*.pcf`（已含中文，但尺寸固定 9–13px，需裁剪/缩放凑 16×16）。
- 现有固件中文点阵精确格式（`display.cpp::drawGlyph16`，323 行）：每个字 = `const uint16_t glyph[16]`，16 行，每行一个 uint16_t；`bit15=最左列, bit0=最右列`，黑色=1（置位）；取位用 `(bits & (1<<(15-col)))`，与字节序无关。
- 生成脚本产物：`const uint16_t cjk16[N][16]`，与现有 drawGlyph16 直接兼容 → 固件绘制代码零改动，只需把写死的工期标签换成查表。原料用 TTF+Pillow(freetype)：渲染 16×16 灰度 → 阈值二值化(黑=1) → 打包 uint16_t → 输出 `.h`。

## 为何必须"预生成字库"——历史显示链路（已探明，2026-07-13）
- 原本项目中文显示有两条链路，设备端几乎不画中文，故烧录不需要字库脚本：
  1. **设备端只硬编码十几个"星期时段"字**：`display.cpp:310 periodGlyphs` 用 `GLYPH_SHEN/GUY_YE/...` 等写死的 `const uint16_t` 数组（深夜/凌晨/早晨/上午/中午/下午/傍晚/夜晚），非从 TTF 生成、非脚本产物。设备端"通用中文渲染"能力为零。
  2. **所有带中文画面由后端实时渲染 BMP 下发**：`backend/core/json_renderer.py:419/1444` 加载 `NotoSerifSC-Regular.ttf`，用 Pillow/freetype 把 JSON 内容渲染成 1-bit BMP，设备经 `/api/render` 拉取直接显示。后端有完整字体引擎，故"字库"活在后端运行时，无需预生成 C 数组、无需脚本。
- **结论**：原本"字库"在后端（运行时 TTF+freetype），不在设备端；设备只是 BMP 显示器 + 少量硬编码字，所以烧录无脚本。现在做"待办原生渲染"= 设备要自绘任意中文 → 首次需设备端通用字库 → MCU 无 freetype → 必须编译前用脚本把 TTF 转 `cjk16.h` 烧入。脚本=这道预生成工序，是原生渲染路线的硬性前提。

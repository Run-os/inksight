- **真根因：'\' 字形注释末尾的反斜杠触发 C 行续接，吞掉 ']' 字形条目**（2026-07-14 最终定位）：
  - `cjk16.h` 第 4355 行（'\' 0x5C 字形）注释写作 `// 0x5C \`，末尾的 `\` 在 C 里是**行续接**，
    使 `//` 注释延续到下一行，把第 4356 行的 ']' (0x5D) 字形条目**整行吞进注释**。
  - 后果：C 编译器看到的 `ascii18_glyphs` 数组只有 **94 项**（缺 ']'），从 ']' 起所有字形错位 +1：
    `ascii18_glyphs[cp-0x20]` 对 'm'(idx77)返回的是 'n'、'i'(idx73)返回 'j'…… 故 "milk"→"njrh"。
    大写/数字在 ']' 之前(idx<61)不受影响，故 "OK42" 正常——完美对应用户现象。
  - 为何排查极久：Python 解析器不懂 C 行续接，把源码数成 95 项（正确），与编译产物的 94 项对不上，
    一度误判为"旧二进制/烧录未生效"。**关键转折**是用"空格+'!'锚点"精确定位数组在 .bin 的真实地址(0x117B4)，
    逐字形解码出 idx77='n'(应为'm')，再回到源码逐行读才发现 '\' 注释末尾的反斜杠。
  - 修复：① `cjk16.h` 第 4355 行注释 `// 0x5C \` → `// 0x5C BS (backslash …)`（不以反斜杠结尾）；
    ② `gen_ascii20.py` 生成器：`disp = "BS" if ch == "\\" else ch`，避免再生成以反斜杠结尾的注释。
  - 验证：重建后从 .bin 解码 idx77 = 'm'(`0x6F3C0`)✓、idx73='i'✓、idx61=']'✓。待真机烧录确认。
  - ⚠️ **可复用教训**：机器生成 C 数组时，行内注释若包含字形的"原始字符"，对 `\`(0x5C) 必须转义/替换，
    否则行末反斜杠会行续接吞掉下一行。同理任何 `// xxx\` 结尾都危险。此 bug 隐蔽性极高（源码肉眼看 95 项都对）。
